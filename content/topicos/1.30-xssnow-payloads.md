## O que é XSS — Cross-Site Scripting

**XSS**, cujo nome completo é Cross-Site Scripting, é um tipo de ataque onde um invasor consegue inserir um mini-programa malicioso dentro de um site legítimo. Quando outra pessoa abre aquela página, o mini-programa executa automaticamente no computador dela, sem que ela perceba nada de errado.

Para entender melhor, pense num quadro de avisos numa escola. Qualquer aluno pode escrever recados nesse quadro. Se um aluno mal-intencionado escreve "Quem ler este aviso, mande uma mensagem com sua senha para mim" e ninguém apaga, todos que passarem pelo corredor lerão o aviso. O "aviso malicioso" é como um ataque XSS: foi inserido num lugar confiável, e agora afeta todos que visitam.

No mundo digital, esse "aviso malicioso" é um código de programação inserido num campo de texto de um site — como a caixa de comentários de um blog, o campo de busca de uma loja online, ou qualquer outro lugar onde o site aceita texto digitado pelo usuário. Se o site não tiver a proteção certa, esse código malicioso é exibido para todos os visitantes seguintes como parte da página.

## Por que isso importa para você?

O XSS está consistentemente na lista dos tipos de ataque mais comuns em sites desde a década de 2000, e continua presente porque é difícil de eliminar completamente. Sites que parecem completamente normais podem ter essa vulnerabilidade escondida em algum campo.

As consequências para usuários comuns são muito concretas. Um ataque XSS bem-sucedido pode roubar automaticamente o "cookie de sessão" — o passe que prova que você está logado num site. Com esse passe, o atacante acessa sua conta sem precisar de senha. Ele pode redirecionar você para um site falso idêntico ao original onde você vai digitar suas credenciais. Pode registrar tudo que você digita naquela página — incluindo senhas e números de cartão de crédito. Pode modificar o que você vê na página, mostrando um número de conta diferente para uma transferência bancária.

Para desenvolvedores e profissionais de segurança, entender o XSS é obrigatório. É uma das brechas mais encontradas em programas de recompensa por bugs, e corrigi-la exige entender exatamente como funciona.

## Os 4 tipos de XSS — cada um com seu grau de perigo

**XSS Refletido:** Funciona através de um link armadilha. O atacante cria um link especialmente modificado e convence a vítima a clicar nele. O código malicioso vai junto no link, o site o processa e exibe o resultado — contaminado — de volta para quem clicou. É como um envelope que ao ser aberto libera uma substância dentro da própria casa de quem o abriu. Afeta apenas quem clicou naquele link específico. Menos perigoso que os outros, mas muito comum em ataques de phishing.

**XSS Armazenado (Stored XSS):** Este é o mais perigoso. O código malicioso é inserido num formulário do site — como a caixa de comentários — e fica guardado no banco de dados. Agora, toda vez que qualquer pessoa visita aquela página, o código malicioso executa automaticamente. Um atacante deixou um comentário malicioso num blog com um milhão de leitores — e agora afeta cada um deles. Pense num veneno que alguém coloca na fonte de água de uma cidade — todos que beberem são afetados.

**XSS Baseado em DOM:** Este tipo é mais técnico. Ele ataca diretamente o que o navegador mostra na tela, sem que o código malicioso precise passar pelo servidor do site. O servidor pode estar completamente limpo e protegido — o ataque acontece do lado do usuário, no navegador. É como contaminar a garrafa de água depois que ela saiu da fábrica, durante o transporte.

**Blind XSS (XSS Cego):** O código malicioso é inserido num campo que só administradores veem — como um formulário de suporte técnico, um log de atividades, ou um painel de administração. O atacante não vê o resultado imediatamente. Ele espera. Quando um administrador abre aquela área do sistema, o código executa no computador do administrador. Como os administradores têm muito mais permissões do que usuários comuns, conseguir comprometer a sessão deles é muito mais valioso para um atacante.

## Como os ataques passam por filtros de proteção

Sites modernos geralmente têm proteções contra XSS — filtros que tentam detectar e bloquear código malicioso nos campos de texto. O **XSSnow** é uma ferramenta que gera variações de ataques projetadas para passar por essas proteções.

Veja como uma proteção pode ser contornada:

O ataque mais simples — que proteções básicas bloqueiam imediatamente:
```html
<script>alert('ataque')</script>
```

Usando uma imagem quebrada para executar o código — às vezes ignora o filtro de scripts:
```html
<img src=x onerror=alert('ataque')>
```

Usando SVG (imagem vetorial) que muitos filtros não verificam:
```html
<svg onload=alert('ataque')>
```

Usando caracteres especiais para confundir o filtro:
```html
<scr\x00ipt>alert('ataque')</scr\x00ipt>
```

Cada proteção tem pontos cegos diferentes. O XSSnow ajuda pesquisadores de segurança a descobrir quais variações passam por quais proteções, para que os desenvolvedores possam fechar essas brechas.

## Como se defender do XSS — o que desenvolvedores precisam fazer

**Output encoding:** Antes de exibir qualquer texto que o usuário digitou, converta os caracteres perigosos em versões inofensivas. O sinal `<` vira `&lt;`, o `>` vira `&gt;`. Assim, mesmo que alguém tente inserir código malicioso, ele aparece como texto comum na tela em vez de ser executado.

**Content Security Policy (CSP):** Uma política de segurança que você configura no servidor que diz ao navegador: "não execute nenhum script que não venha dos endereços que eu autorizo." Mesmo que um atacante consiga inserir código malicioso na página, o navegador se recusa a executá-lo.

**Frameworks modernos:** Frameworks de desenvolvimento web modernos como React, Vue e Angular implementam automaticamente proteções contra XSS em toda saída de texto. Ao usar esses frameworks, você está protegido por padrão — mas apenas se usá-los corretamente e não "desligar" as proteções manualmente.

1. Um pesquisador de segurança está fazendo bug bounty num grande site de e-commerce. O site tem uma caixa de busca que aceita texto. Ele usa o XSSnow para gerar 50 variações diferentes de ataques XSS e testa cada uma na caixa de busca, observando o que acontece. Após 30 testes, uma variação específica usando uma tag SVG não é filtrada e executa no navegador. Isso significa que qualquer um poderia criar um link especial que, quando clicado por um cliente logado, roubaria o cookie de sessão desse cliente — dando ao atacante acesso completo à conta dele, incluindo dados de cartão de crédito salvos. O pesquisador documenta com capturas de tela, reporta ao programa de bug bounty do site e recebe $2.500 pela descoberta. O site corrige a brecha em 48 horas.

2. Uma equipe de desenvolvimento está prestes a lançar um novo site de avaliações de restaurantes. Um dos desenvolvedores mais experientes propõe fazer um teste de XSS na seção de comentários antes de abrir para o público. Eles usam o XSSnow para testar diferentes ataques na caixa de comentários. Descobrem que o código malicioso básico é bloqueado, mas uma variação com caracteres Unicode especiais passa pelos filtros e é armazenada no banco de dados. Se tivessem lançado sem o teste, qualquer pessoa poderia ter deixado um comentário malicioso que afetaria todos os visitantes seguintes. A correção leva 2 horas — antes de qualquer usuário real ser afetado.

3. Um gerente de um pequeno banco digital percebe que clientes estão reclamando que suas contas foram acessadas sem autorização, mesmo tendo senhas fortes. Uma auditoria de segurança aponta para um campo no sistema de mensagens interno do banco — onde clientes enviam dúvidas para o suporte. Esse campo não tinha proteção contra XSS. Um atacante havia enviado uma mensagem com código malicioso que, quando o atendente de suporte abria para responder, executava no computador do atendente e capturava o cookie de sessão administrativo. Com esse cookie, o atacante acessava as contas dos clientes pelo painel de administração. O banco implementa proteção de output encoding e CSP, e os ataques param imediatamente. O prejuízo foi de R$ 340.000 em transferências fraudulentas — que poderia ter sido zero com proteção adequada desde o início.
