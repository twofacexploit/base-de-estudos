## O autor do case

**Souhaib Naceri**, conhecido online como **h4x0r_dz**, é um pesquisador de segurança argelino que se especializou em algo que poucos fazem: em vez de parar quando encontra um problema pequeno, ele continua investigando para descobrir se aquele problema pequeno abre caminho para problemas maiores. Essa habilidade de "encadear" vulnerabilidades — onde uma falha leva à próxima, que leva a uma terceira — é o que o tornou reconhecido na comunidade de bug bounty.

Este case específico resultou em **$23.000 dólares** de recompensa, o equivalente a aproximadamente R$115.000 na época. O que torna o case educativo não é o valor, mas o raciocínio por trás de cada descoberta e como cada peça se encaixou na seguinte.

## A lição central antes de começar

Problemas isolados costumam ser avaliados como de baixo impacto e pagam pouco — às vezes apenas algumas centenas de dólares. O que Naceri demonstrou neste caso é que três problemas pequenos, quando se conectam numa sequência lógica, formam um ataque de impacto muito maior do que a soma das partes.

É como descobrir que a fechadura da porta dos fundos de um banco é fraca (problema 1), que a porta cofre dentro do banco pode ser aberta de dentro sem código (problema 2), e que o sistema de câmeras tem um ponto cego naquele corredor (problema 3). Cada um sozinho é uma preocupação. Os três juntos são um plano de roubo.

## Vulnerabilidade 1 — Manipulação do "crachá" digital (JWT) — $3.000

Para entender esta primeira falha, você precisa entender o que é um **JWT** (JSON Web Token, ou "Token de Acesso Web"). Pense assim: quando você faz login num site, o site precisa saber quem você é em cada página que você visita, sem te pedir a senha de novo a cada clique. Para isso, ele te dá um "crachá digital" — um pequeno arquivo de texto criptografado que você carrega no seu navegador e apresenta automaticamente para o site em cada requisição. Esse crachá diz: "esse usuário é o João, com cargo de Cliente Normal, com permissão de fazer X e Y".

O **JWT** é o formato mais comum desse crachá digital. Ele tem três partes: informações do usuário, permissões e assinatura de segurança. A assinatura existe para garantir que ninguém alterou as informações — se você tentar modificar o crachá, a assinatura fica inválida.

A falha que Naceri encontrou: o sistema em questão verificava a assinatura do JWT, mas **não validava corretamente o conteúdo do campo "realm"** (um campo que indicava qual área do sistema o usuário deveria acessar). Ele descobriu que podia modificar esse campo específico e o sistema aceitava sem questionar, mesmo com a assinatura técnica inválida.

```
Login normal:  {"realm": "test-user"}      → Acesso à área de usuário comum
Com a falha:   {"realm": "test-dashboard"} → Acesso ao painel administrativo
```

Em linguagem simples: era como se o segurança do shopping verificasse a foto do crachá mas não lesse o cargo escrito nele. Naceri apagou "Visitante" e escreveu "Diretor" — e passou sem problema.

O impacto isolado desse problema: acesso ao painel administrativo de um único sistema de testes. Pagamento: **$3.000**. Naceri poderia ter parado aqui. Não parou.

## Vulnerabilidade 2 — Porta dos fundos no sistema de autenticação — $20.000

Com acesso ao painel de testes, Naceri continuou investigando e encontrou algo muito mais grave no sistema principal de autenticação.

Para entender: quando você faz uma requisição para um site que exige login, você envia suas credenciais com um cabeçalho (uma linha de informação invisível) que diz ao servidor como interpretar o que você está enviando. Normalmente esse cabeçalho é: `Authorization: Bearer TOKEN` — onde "Bearer" é uma palavra técnica que indica o tipo de credencial que está sendo enviada.

A falha: o sistema verificava autenticação quando recebia o cabeçalho completo `Authorization: Bearer TOKEN`. Mas quando Naceri enviou a requisição **sem a palavra "Bearer"**, apenas `Authorization: TOKEN`, o sistema simplesmente deixou passar — sem verificar nada.

```
Caminho normal:     Authorization: Bearer TOKEN_VALIDO  → Sistema verifica e controla
Caminho com falha:  Authorization: TOKEN_QUALQUER       → Sistema não verifica, libera
```

É o equivalente a uma cancela de estacionamento que verifica o ticket quando você coloca no leitor correto — mas tem um segundo encaixe que ninguém configurou e que aceita qualquer cartão sem verificar. Quem souber do segundo encaixe entra sem pagar e sem registro.

O impacto: esse não era mais um sistema de testes. Era o sistema de autenticação principal. Qualquer pessoa que soubesse dessa técnica poderia acessar qualquer conta, sem senha, no sistema inteiro. A combinação desta falha com a primeira — que já dava acesso administrativo — resultou em acesso irrestrito total ao sistema. Pagamento adicional: **$20.000** (compartilhado com um pesquisador que havia reportado parcialmente antes).

## Vulnerabilidade 3 — Upload de arquivo sem controle de destino

A terceira vulnerabilidade foi descoberta durante a exploração das anteriores. O sistema permitia que usuários fizessem upload (envio) de arquivos. O que Naceri descobriu é que ao enviar um arquivo, ele podia especificar **para qual pasta dentro do servidor** o arquivo devia ser salvo — e o sistema aceitava qualquer destino sem questionar.

Isso é extremamente grave. Imagine que um site tem um servidor onde ficam guardados:
- O código JavaScript que todos os usuários baixam quando acessam o site
- Os instaladores de aplicativos que os usuários baixam
- Documentos oficiais publicados para download

Com essa falha, Naceri podia substituir qualquer um desses arquivos pelo que quisesse. O impacto teórico:

**Substituir o JavaScript do site:** Todos os usuários que acessassem o site baixariam automaticamente um código malicioso em vez do código legítimo. Esse código poderia roubar senhas, dados de cartão, informações pessoais — de todos os visitantes, sem que percebessem.

**Substituir arquivos de instalação:** Usuários que baixassem o aplicativo oficial estariam baixando um vírus assinado pelo certificado legítimo da empresa.

**Substituir documentos oficiais:** Documentos de contrato, termos de serviço, faturas — poderiam ser alterados silenciosamente.

Resultado: um atacante com essas três vulnerabilidades controlava completamente o site e poderia causar dano a todos os seus usuários simultaneamente.

## As ferramentas que tornaram tudo possível

Cada ferramenta cumpriu um papel específico nessa investigação:

**Burp Suite** — O intermediário entre o navegador e o servidor. Naceri usou para interceptar e modificar cada requisição, testando variações sem que o site soubesse. É como gravar cada conversa e poder reenviar com palavras trocadas.

**ffuf** — Ferramenta que testa automaticamente centenas de endereços possíveis em busca de páginas e painéis escondidos. Foi assim que o painel administrativo foi descoberto — ele existia mas não estava linkado em nenhum lugar visível.

**Wayback Machine** — Arquivo histórico da internet que guarda versões antigas de sites. Naceri usou para encontrar páginas que foram removidas do site mas ainda estavam acessíveis nos servidores — "fantasmas" de versões antigas com configurações diferentes.

**JWT.io** — Ferramenta online para ler, modificar e gerar tokens JWT. Foi a ferramenta usada para decodificar o crachá digital, entender sua estrutura e testar as modificações no campo "realm".

## A plataforma e o pagamento

Todo o processo foi documentado e enviado pelo **HackerOne** — a maior plataforma de bug bounty do mundo. A empresa avaliou o impacto combinado das três vulnerabilidades e fez o pagamento de **$23.000 no total**, distribuídos entre as descobertas de acordo com o impacto de cada uma.

O relatório enviado descrevia cada vulnerabilidade separadamente, com passos detalhados para reproduzir, evidências (capturas de tela de cada etapa), e uma seção especial mostrando como as três se conectavam para criar um impacto maior que qualquer uma sozinha.

## A moral do case — o que separa o bom do excelente

Um pesquisador médio teria encontrado a primeira vulnerabilidade, reportado por $3.000 e seguido em frente. Naceri fez a pergunta que os melhores sempre fazem: "Se isso existe, o que mais pode existir por perto?"

A resposta a essa pergunta transformou um achado de $3.000 em um de $23.000 — 7,6 vezes mais. Não por sorte, mas por método: a disciplina de não parar na primeira descoberta e de ver falhas como pontos de entrada para investigação mais profunda.

## Exemplos do dia a dia

1. Pense no sistema de uma academia de ginástica. O crachá de membro (equivalente ao JWT) diz "João — plano básico — acesso às esteiras". A primeira falha seria descobrir que você pode alterar o crachá para dizer "João — plano premium — acesso a tudo". A segunda falha seria descobrir que a catraca da porta VIP aceita qualquer crachá que não contenha a palavra "básico" — então você remove a informação de plano e entra. A terceira seria descobrir que no vestiário VIP existe um painel de controle onde você pode alterar as informações de todos os membros do sistema. As três juntas dariam controle total sobre a academia. Separadas, a primeira é um transtorno, a segunda é uma brecha pontual, e a terceira sem as outras duas seria difícil de alcançar.

2. Um pesquisador iniciante encontra apenas a primeira vulnerabilidade deste case e recebe $500 porque não soube identificar o impacto completo — parou quando viu "acesso ao painel de testes" e não investigou o que havia dentro. Naceri encontrou os três problemas, demonstrou como cada um preparava o caminho para o seguinte, quantificou o impacto potencial em usuários reais e recebeu $23.000 — 46 vezes mais. A diferença não foi sorte ou habilidade técnica superior. Foi método: a disciplina de continuar investigando quando outros pararam.
