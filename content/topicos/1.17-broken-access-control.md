## A base: autenticação vs autorização

Duas palavras parecidas que significam coisas diferentes:

```
Autenticação = QUEM é você? (login, senha, biometria)
Autorização  = O QUE você pode fazer? (permissões, papéis)
```

**Broken Access Control** é falha na **autorização** — você fez login legitimamente, mas o sistema deixa você fazer coisas que não devia.

## Os 7 tipos de exploração

| # | Técnica | Como funciona |
|---|---|---|
| 1 | **IDOR** | Troca ID na URL: `/pedidos/1234` → `/pedidos/1235` |
| 2 | **Method Discrepancy** | GET retorna 403, POST retorna 200 no mesmo endpoint |
| 3 | **HTTP Parameter Pollution** | Duplica parâmetro: `?userId=meu&userId=victim` |
| 4 | **Static Keywords** | Troca "my" por ID real: `/workspace/my` → `/workspace/org123` |
| 5 | **JWT Flaws** | Manipula claims: `role="user"` → `role="admin"` |
| 6 | **Logic Flaws** | Fluxo multi-step perde o controle de autorização no meio do caminho |
| 7 | **Second-Order** | Falha aparece em contexto secundário (middleware, background job) |

## CIA Triad — como avaliar o impacto

Toda falha de segurança é classificada pelo que ela quebra:

- **C = Confidencialidade** — ver dados que não deveria (ex: ler pedidos de outros)
- **I = Integridade** — modificar dados que não deveria (ex: mudar preços)
- **A = Disponibilidade** — derrubar o serviço (ex: apagar conteúdo alheio)

Bug que quebra os 3 = **crítico** = paga muito bem.

## Por que é #1

É a vulnerabilidade **mais comum** na lista OWASP Top 10. Quase todo site tem alguma — basta olhar com atenção. E diferente de XSS ou SQLi, não tem "fórmula mágica" pra defender: exige arquitetura bem pensada.
