## O que é Broken Access Control

Para entender esta vulnerabilidade, você precisa primeiro entender a diferença entre duas palavras que parecem parecidas mas significam coisas completamente diferentes: **autenticação** e **autorização**.

**Autenticação** é provar quem você é. É a etapa do login: você fornece seu usuário e senha, o sistema verifica, e confirma "sim, você é quem diz ser". É como mostrar seu RG na entrada de um evento — você prova sua identidade.

**Autorização** é o que você tem permissão de fazer depois de entrar. É diferente. No mesmo evento, você pode entrar como convidado comum (acesso à área geral) ou como VIP (acesso ao camarote). Ambos provaram sua identidade — mas têm permissões diferentes.

O **Broken Access Control** (em português: "Controle de Acesso Quebrado") é uma falha na **autorização**. O sistema sabe quem você é — você fez login corretamente — mas deixa você fazer coisas que não deveria. É como entrar num hotel com sua chave, e descobrir que ela abre todos os outros quartos também, não só o seu.

Esta é a vulnerabilidade classificada como **número 1** no OWASP Top 10 — o ranking global das falhas de segurança mais comuns na internet, publicado por um dos grupos de especialistas mais respeitados da área. Quase todo site tem alguma variação disso. E diferente de outros tipos de ataque, não existe um remédio simples — exige que o sistema inteiro seja projetado com cuidado desde o início.

## Por que essa vulnerabilidade é tão comum

A razão é simples: é muito mais fácil implementar autenticação do que autorização. Autenticação tem uma resposta binária — você está logado ou não. Autorização é infinitamente mais complexa: você pode ver esse arquivo, mas não aquele. Pode editar seu perfil, mas não o de outro. Pode ver seus pedidos, mas não os pedidos de clientes. Pode pagar uma conta, mas não cancelar a conta de outra pessoa.

Cada permissão dessas precisa ser verificada separadamente, em cada página, em cada operação, para cada usuário. Um sistema com mil funcionalidades tem potencialmente mil verificações de autorização. Se uma delas for esquecida, existe uma brecha.

Desenvolvedores sob pressão de prazo, times pequenos sem especialista em segurança, e sistemas que cresceram organicamente sem planejamento são os ambientes mais férteis para essa falha aparecer.

## Os 7 tipos de exploração — com exemplos concretos

### Tipo 1 — IDOR (Insecure Direct Object Reference)

**O que é:** O sistema expõe na URL (o endereço da página) um identificador direto de um objeto — como um número de pedido, ID de usuário ou código de documento. E não verifica se o usuário que está pedindo tem permissão de ver aquele objeto específico.

**Como explorar:** Você acessa seu pedido em `/pedidos/5482`. Por curiosidade, troca para `/pedidos/5481`. O sistema retorna o pedido completo de outra pessoa — nome, endereço, CPF, produtos comprados.

**Na vida real:** Você já usou um e-commerce onde o número do pedido aparece na URL? Se o sistema não verificar se aquele pedido pertence a você, qualquer pessoa logada pode ver os pedidos de qualquer outro cliente simplesmente trocando o número.

**Por que é grave:** IDOR é a falha de Broken Access Control mais comum e mais fácil de explorar. Não exige nenhuma ferramenta especial — só um navegador e curiosidade.

### Tipo 2 — Diferença de método HTTP

**O que é:** O sistema protege uma funcionalidade quando acessada de uma forma (método GET — que é quando você simplesmente visita uma página), mas não protege quando acessada de outra forma (método POST — que é quando você envia um formulário, ou método PUT/DELETE — que são usados por aplicativos).

**Como explorar:** Tentar acessar `/admin/usuarios` via GET é bloqueado. Tentar via POST não é. O sistema verificou o caminho mas não o método.

**Na vida real:** É como uma portaria que verifica documentos para quem entra pela porta principal, mas não tem segurança na entrada de serviço. Quem conhece a entrada de serviço passa sem ser verificado.

### Tipo 3 — Duplicação de parâmetro

**O que é:** O sistema recebe informações e, quando recebe o mesmo campo duas vezes, processa o segundo e ignora o primeiro — ou faz o contrário. Um atacante explora isso enviando o campo duas vezes com valores diferentes.

**Como explorar:** Um formulário tem o campo `cargo=funcionario`. O atacante envia `cargo=funcionario&cargo=administrador`. O sistema confundido usa o segundo valor e eleva o privilégio.

**Na vida real:** É como enviar um formulário de banco com o campo "valor da transferência" escrito duas vezes, com valores diferentes, para confundir o sistema sobre qual valor processar.

### Tipo 4 — Troca de palavras fixas por identificadores reais

**O que é:** O sistema usa a palavra "me" (eu, meu) na URL para representar o usuário atual. Por exemplo: `/api/workspace/me` retorna os dados do usuário logado. A falha é quando trocar "me" por um ID real (`/api/workspace/empresa123`) também funciona — e não verifica permissão.

**Como explorar:** Você acessa `/workspace/me` normalmente. Descobre que também pode acessar `/workspace/qualquer-empresa` e ver dados de organizações que não são suas.

**Na vida real:** É como um banco onde você pode ver seu extrato em "minha conta", mas descobrir que pode também ver o extrato de qualquer outra conta só substituindo a palavra "minha" por um número de conta.

### Tipo 5 — Manipulação do JWT (o crachá digital)

**O que é:** O JWT é o "crachá digital" que o servidor emite quando você faz login. Ele contém informações sobre você e suas permissões. A falha acontece quando o servidor não verifica adequadamente a assinatura do JWT — permitindo que você altere seu cargo, seu ID, ou suas permissões no crachá.

**Como explorar:** Você faz login como usuário comum. Pega seu JWT, decodifica (usando o site jwt.io, por exemplo), troca `"cargo": "usuario"` para `"cargo": "administrador"`, e reenvia. Se o servidor aceitar sem verificar a assinatura, você agora é administrador.

**Na vida real:** É o equivalente a encontrar um crachá de empresa na rua, raspar o nome e o cargo, escrever seu próprio nome e "Diretor", e conseguir entrar em qualquer área do prédio porque o segurança verifica o crachá mas não a foto nem o sistema de autenticação.

**Vimos isso no tópico 1.14** — foi exatamente essa falha que Souhaib Naceri explorou no case dos $23.000.

### Tipo 6 — Falha de lógica de negócio em processos multi-etapas

**O que é:** Processos com várias etapas (como uma compra online: carrinho → endereço → pagamento → confirmação) às vezes verificam permissão nas primeiras etapas mas esquecem de verificar nas etapas do meio ou do final. Um atacante que pula diretamente para uma etapa avançada pode ter acesso a funcionalidades que não deveria.

**Como explorar:** Um processo de aprovação de crédito tem 5 etapas. As etapas 1 e 2 verificam identidade. A etapa 5, que emite o contrato, verifica se a etapa 4 foi completada. Mas não verifica se as etapas 1, 2 e 3 foram feitas pelo mesmo usuário. Alguém pode completar as etapas iniciais com dados reais, e na etapa final injetar dados falsos que não passariam pela verificação inicial.

**Na vida real:** É como uma fila de banco onde o segurança confere seu documento na entrada. Se você sair da fila e voltar pela metade, o caixa pode assumir que você já foi verificado — sem checar de novo.

### Tipo 7 — Injeção de segunda ordem

**O que é:** A falha mais difícil de detectar. O problema é criado em um lugar do sistema, mas o impacto aparece em outro lugar, muito depois. O sistema pode sanitizar (limpar) a entrada em um ponto, mas usar esses dados sem verificação em outro contexto.

**Como explorar:** Você cria um nome de usuário que contém um comando malicioso: `admin'--`. O sistema armazena isso no banco de dados de forma segura. Mas quando um administrador busca o seu nome num relatório diferente, o sistema usa o nome armazenado numa consulta diferente, sem a mesma proteção — e o comando malicioso executa.

**Na vida real:** É como deixar uma mensagem num recado que parece inocente, mas que quando for lido em outro contexto, instrui alguém a fazer algo que não deveria.

## CIA Triad — como avaliar o impacto de uma falha

Quando um pesquisador de segurança encontra uma falha de Broken Access Control, ele avalia o impacto usando três dimensões chamadas de **CIA** (não a agência de espionagem — é uma sigla técnica da área de segurança):

**C — Confidencialidade (Confidentiality):** A falha permite que alguém **veja** dados que não deveria? Exemplo: ver o pedido de outro cliente. Viola a confidencialidade.

**I — Integridade (Integrity):** A falha permite que alguém **altere** dados que não deveria? Exemplo: modificar o preço de um produto, alterar o endereço de entrega de outro cliente. Viola a integridade.

**A — Disponibilidade (Availability):** A falha permite que alguém **apague ou bloqueie** dados e serviços? Exemplo: apagar o conteúdo do perfil de outro usuário, derrubar o sistema inteiro. Viola a disponibilidade.

Uma falha que viola apenas um dos três já é séria. Uma falha que viola os três ao mesmo tempo é classificada como **crítica** e rende as maiores recompensas em programas de bug bounty — às vezes dezenas ou centenas de milhares de dólares.

## Por que o Broken Access Control é o número 1

Segundo o **OWASP Top 10** — o ranking anual das 10 vulnerabilidades mais comuns e impactantes na web — o Broken Access Control é a falha mais presente. Ela superou Injeção SQL (que era historicamente número 1) justamente porque:

É silenciosa: diferente de um sistema que cai ou de um erro visível, o acesso indevido muitas vezes não deixa rastro óbvio. O usuário acessa, vê o que não deveria, e vai embora. Sem alarme.

É ubíqua: qualquer sistema com múltiplos níveis de usuário potencialmente tem Broken Access Control em algum lugar. A ausência de verificações específicas é muito mais comum do que se imagina.

É difícil de detectar automaticamente: scanners automáticos de segurança são ótimos para detectar injeção SQL, XSS e outras falhas técnicas. Broken Access Control muitas vezes exige raciocínio humano — entender o que cada usuário deveria poder fazer e testar se o sistema respeita essas regras.

## Por que isso importa para você?

Você usa sistemas com múltiplos usuários todos os dias: seu banco, o portal de saúde do plano médico, o sistema de notas da escola dos filhos, o e-commerce onde compra. Todos esses sistemas guardam seus dados e dados de milhares de outras pessoas. Se qualquer um deles tem Broken Access Control, seus dados podem estar acessíveis a outros usuários — sem que você ou o sistema saiba.

Como consumidor, você pode verificar o básico: se ao mudar o número de um pedido na URL você vê dados de outro cliente, reportar isso ao SAC da empresa (é um serviço importante que você está prestando). Como alguém que desenvolve qualquer tipo de sistema digital, entender Broken Access Control é fundamental — é a falha que mais aparece em auditorias e a que mais envergonha empresas quando vaza.

## Exemplos do dia a dia

1. Você faz uma compra em uma loja virtual e recebe o email de confirmação com o link para acompanhar seu pedido: `https://loja.com.br/pedidos/5482`. Você clica, vê seus dados normalmente. Por curiosidade, troca o número para 5481 na barra de endereços do navegador. A página carrega normalmente — com o nome completo, CPF, endereço e lista de produtos comprados de outra pessoa. Você acabou de descobrir um IDOR. O correto agora é reportar ao SAC da empresa com o print — não usar a informação de forma alguma. Esse tipo de falha já resultou em multas milionárias de órgãos regulatórios europeus e americanos contra empresas que negligenciaram a proteção.

2. Um funcionário recém-contratado de uma empresa descobre que o sistema interno de gestão tem uma URL de administração: `https://sistema.empresa.com/administracao`. Quando tenta acessar, é redirecionado para a página principal — parece bloqueado. Mas por curiosidade, ele adiciona manualmente o endereço e consegue entrar. O sistema verificou se ele estava logado (sim, estava), mas esqueceu de verificar se ele tinha permissão de administrador. Ele agora tem acesso a salários de todos os funcionários, configurações do sistema e dados de clientes que jamais deveria ver. Um funcionário mal-intencionado nessa situação poderia causar danos imensos. Um funcionário honesto vai reportar ao departamento de TI imediatamente.

3. Uma plataforma de telemedicina permite que médicos acessem prontuários de seus pacientes. O endereço do prontuário é: `/prontuario/12345`. Um médico descobre que trocando o número do prontuário consegue ver prontuários de pacientes que nunca foram seus — incluindo histórico completo de saúde, diagnósticos psiquiátricos, resultados de exames. Isso é uma violação gravíssima de dados de saúde, regulada por leis específicas como a LGPD no Brasil. A plataforma poderia enfrentar processos, multas regulatórias e perda de credibilidade — tudo por uma verificação esquecida que custaria poucas linhas de código para ser implementada.
