## A estrutura

```
projeto/
├── CLAUDE.md          ← memória e regras do projeto
├── DESIGN.md          ← identidade visual (ver 6.4)
├── docs/
│   ├── decisions/     ← por que cada decisão foi tomada
│   └── runbooks/      ← guias de operação
├── .claude/
│   ├── hooks/         ← verificações automáticas
│   └── skills/        ← workflows reutilizáveis
│       ├── code-review/SKILL.md
│       ├── refactor/SKILL.md
│       └── release/SKILL.md
└── src/
    └── api/CLAUDE.md  ← instruções específicas do módulo
```

## Os arquivos raiz

- **CLAUDE.md** — o "manual" que o agente lê sempre. Regras do projeto, comandos importantes, convenções.
- **DESIGN.md** — identidade visual (cores, tipografia, espaçamentos) — aplicado automaticamente em tudo que tem UI.

## O diretório `.claude/`

- **hooks/** — scripts que rodam automaticamente em eventos (antes de commitar, depois de salvar, etc). Exemplos: rodar lint, verificar secrets, formatar código.
- **skills/** — workflows que o Claude pode invocar. Cada skill tem seu próprio SKILL.md.

## CLAUDE.md por módulo

Cada subdiretório importante pode ter seu próprio `CLAUDE.md`. Quando o Claude trabalha em `src/api/`, ele lê o `src/api/CLAUDE.md` com regras específicas do módulo (ex: "use sempre Drizzle, nunca queries SQL cruas").

## O loop de melhoria

```
Claude erra → você adiciona regra ao CLAUDE.md
Você se repete → você cria uma skill
Algo quebra → você adiciona um hook
= Claude fica mais inteligente continuamente
```

É um processo **vivo**: cada erro vira regra nova, cada repetição vira skill, cada bug vira hook. Com o tempo, o projeto se torna difícil de estragar — porque todas as armadilhas já viraram guardas.

## Por que isso muda tudo

Antes: cada projeto novo começa do zero com o Claude. Agora: você acumula inteligência específica do projeto, que **persiste** e **melhora com o uso**. O investimento é front-loaded, mas a produtividade composta é enorme.
